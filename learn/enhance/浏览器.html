# 客服端存储

localStorage/sessionStorage 长久储存/窗口关闭删除， 体积限制为 4~5M
直接访问：localStorage.属性
方法：getItem/setItem/removeItem/clear

onstorage事件 IE9+
属性：key/newValue/oldValue/storageArea/url

cookies http 中自动携带， 体积上限为 4K， 可自行设置过期时间
字符串形式，键值对，分号隔开，存储前用 encodeURIComponent 编码，读取时用 decodeURIComponent 解码，同时提供一个 max-age 属性指定有效期（单位 s）
要设置 cookie 的 path、domain、secure 属性，在 cookie 值前设置

标准不允许浏览器保存超过 300 个 cookie（现代浏览器允许超过 300 个），为每个 web 服务器保存不超过 20 个 cookie，每个 cookie 数据不超过 4KB

$.ajax()提交 new FormData()时，需要 contentType: false, processData: false

# 应用程序存储

在html标签中设置manifest属性，指向清单文件，扩展名为.appcache
清单文件必须以CACHE MANIFEST开头，其余是URL列表，一行一个URL，可以用#来注释该行
复杂清单：
CACHE MANIFEST
# 默认区域，需要缓存的
CACHE:
main.js
# 每行包含两个URL，第一个是URL前缀，任何能匹配到该前缀的URL都不会被缓存，第二个是缓存资源；从网络中获取，失败则使用缓存资源
FALLBACK:

# 资源从不缓存
NETWORK:

缓存更新：
CACHE MANIFEST
# main version 1 (更改这个值，让浏览器更新这个文件)
main.js

applicationCache对象可以进行监控
事件：
onchcking  当应用程序载入时，都会检查该清单文件
onnoupdate  清单文件没有改动，应用程序也已经缓存了
ondownloading 还未缓存应用程序，或清单文件有改动
onprogress  下载过程中
oncached  下载完成并首次将应用程序下载到缓存中时
onupdateready  下载完成并将缓存中的应用程序更新后
onerror  处于离线状态，检查清单列表失败
onobsolete  引用不存在的清单文件，将应用从缓存中移除
方法：
update() 显示调用检查更新
swapCache() 通知浏览器弃用之前的缓存，都从最新缓存中获取
属性：
UNCACHED  应用程序没有设置manifest属性，未缓存
IDLE  清单文件已经检查完毕，并已经缓存了最新的应用程序
CHECKING  正在检查清单文件
DOWNLOADING  正在下载并缓存清单中的所有文件
UPDATEREADY  下载和缓存最新版的应用程序
OBSOLETE  清单文件不存在，缓存将被清除

# webWorker

webWorker 用于处理大量的计算，使得页面不会出现卡顿。

  • 创建 Worker 时，JS 引擎向浏览器申请开一个子线程（子线程是浏览器开的，完全受主线程控制，而且不能操作 DOM）
  • JS 引擎线程与 worker 线程间通过特定的方式通信（postMessage API，需要通过序列化对象来与线程交互特定的数据）
